# AR Floor Plan Demo — Project Handoff

> **For Claude Code:** This document is the handoff from a claude.ai conversation. Read this first before doing anything. The user is Dr. Salar Kamari (USM, Construction Management). The goal of this project is a no-install web AR demo for his AI in Construction course / BIM work.

---

## Project goal

Build a **web-based AR demo** that:
- Requires **no app install** — user scans a QR code, opens a web page, and AR runs in the browser
- Overlays a **3D building model** onto a **2D floor plan** (printed or shown on a screen)
- Has **zero backend compute** — everything runs client-side on the user's phone
- Is **hosted free** on GitHub Pages (HTTPS required for camera access)

Use case: classroom demo for AEC 450 (BIM in Construction) and broader "AI in Construction" course materials. Students scan a QR → see a 3D building rise out of a printed floor plan in AR.

## Tech stack chosen

- **MindAR** (image-target AR library) — chosen over AR.js for better image tracking quality
- **Three.js** (3D rendering) — version 0.160.0
- **Pure HTML/CSS/JS** — no build step, no bundler, no framework
- **GitHub Pages** for static hosting (HTTPS auto, free)

## Why these choices

1. **No app install**: WebXR / camera + canvas works in mobile Safari and Chrome over HTTPS. No App Store / Play Store friction for students.
2. **MindAR over AR.js**: more robust image tracking, better feature point detection on detailed drawings like floor plans.
3. **Static hosting**: Floor plans and 3D models are pre-generated. No server-side compute needed. GitHub Pages is free and gives stable HTTPS.
4. **Procedural placeholder model**: real `.glb` not provided yet — built a programmatic Three.js building that roughly matches the placeholder floor plan footprint. Easy to swap out for a real model later.

### File responsibilities

**`index.html`** (the main deliverable)
- Landing screen with editorial design aesthetic (paper-textured background, Georgia serif headings, red `#d94f30` accent)
- Embedded SVG preview of the floor plan
- "Launch AR Experience" button kicks off MindAR
- AR view: full-screen camera + Three.js overlay
- Procedural 3D building defined in `createBuilding()` function — L-shaped 2-story building with red accent tower, glass curtain wall, entrance canopy
- Smooth rise-in animation when target is detected
- Exit button returns to landing

**`compiler.html`** (setup tool, run once)
- Renders the embedded SVG to a 1600×1200 PNG
- Uses MindAR's image compiler (runs in browser) to extract feature points and produce a `.mind` binary
- User downloads `targets.mind` and places it next to `index.html`
- This is needed because MindAR cannot compile tracking targets at runtime — pre-compilation is required

**`targets.mind`** (generated, not in source)
- Binary file MindAR loads at runtime to know what to track
- Must be regenerated whenever the floor plan SVG changes
- ~100–500 KB depending on image complexity

## Critical implementation details (don't break these)

### 1. The SVG floor plan is duplicated in TWO files
The same `<svg viewBox="0 0 800 600">` block appears in both `index.html` (as preview on landing) and `compiler.html` (as compilation source). **If you change one, change the other** or re-run the compiler.

### 2. Coordinate mapping (SVG → 3D)
The procedural building in `createBuilding()` uses these constants:
```js
const SCALE_X = 1.0 / 800;  // SVG width → 1.0 units in 3D
const SCALE_Y = 0.75 / 600; // SVG height → 0.75 units in 3D
```
Building blocks are positioned with helper `block(svgX, svgY, svgW, svgH, height3D, mat)` which converts top-left SVG coords to centered 3D coords. **The 3D origin is the center of the floor plan image.**

### 3. MindAR axis convention
MindAR's image-target plane: x right, **y up (out of the page)**, z in-plane. So the building extrudes along **+Y**. Don't confuse this with Three.js's default convention where the image plane would be XY and extrusion would be along Z.

### 4. Import map for ESM
The page uses `<script type="importmap">` to resolve `three` and `mindar-image-three` from jsDelivr CDN. **Don't switch to non-module script tags** — MindAR's three.js binding requires the ESM build.

### 5. Camera permission requires HTTPS
GitHub Pages provides this automatically. `localhost` works too for local dev. Plain HTTP will silently fail to start the camera.

### 6. Target tracking quality
The current placeholder floor plan has been deliberately enriched with:
- Asymmetric L-shape building outline
- Red `A`–`E` corner markers with circles
- Dimension lines and numbers
- Italic Georgia room labels
- Furniture/fixture details
- A title block and compass

This is because **clean architectural drawings track poorly** — they lack distinctive feature points. When swapping in Salar's real floor plans, if tracking is unreliable, the fix is usually: add a unique border, a logo, distinctive text, or asymmetric markers somewhere on the sheet.

## Deployment workflow (already established)

User's plan: deploy to GitHub Pages. Steps:

1. Create public repo
2. Push `index.html` + `targets.mind` (and optionally `compiler.html`, `floorplan.png`, `README.md`)
3. Settings → Pages → enable on `main` branch, root folder
4. URL: `https://<username>.github.io/<repo>/`
5. Feed URL into any QR generator → print → done

To regenerate `targets.mind`: open `compiler.html` locally in any browser, click through the 4 buttons, download the file, replace in the repo.

## Likely next iterations (anticipate these)

In rough priority order based on Salar's typical workflow:

### A. Replace placeholder floor plan with a real one
Most likely first request. Salar will provide a real floor plan PNG/SVG (likely exported from Revit or AutoCAD).
- Replace the `<svg>` in `index.html` (the preview) with `<img src="real-floorplan.png">` OR embed the new SVG
- Replace the SVG in `compiler.html` similarly (or load the PNG directly)
- Re-run compiler → get new `targets.mind`
- **Verify tracking quality** — if poor, add asymmetric markers or a distinctive border

### B. Replace procedural building with a real .glb model
Second most likely. Salar has Revit models that can be exported to FBX → GLB via Blender, or directly from Revit via plugins.

Modify `index.html`:
```js
// Add to imports
import { GLTFLoader } from 'three/addons/loaders/GLTFLoader.js';

// Replace createBuilding() with async version
async function createBuilding() {
  const loader = new GLTFLoader();
  const gltf = await loader.loadAsync('./building.glb');
  const group = gltf.scene;
  group.scale.set(0.5, 0.5, 0.5);  // tune to match floor plan
  group.userData.targetY = 0;
  group.userData.currentY = -0.3;
  group.position.y = -0.3;
  // Add lights
  group.add(new THREE.AmbientLight(0xffffff, 0.55));
  const sun = new THREE.DirectionalLight(0xfff4e0, 1.0);
  sun.position.set(2, 4, 3);
  group.add(sun);
  return group;
}
```
Also make `startAR()` await `createBuilding()` since it'll now be async.

### C. Scale / align the model to the floor plan
After swapping in a real model, it will almost certainly be wrong size, wrong rotation, wrong origin. Likely tweaks:
- `group.scale.set(s, s, s)` — typically 0.001 to 0.5 depending on model units (Revit exports often in mm or feet)
- `group.rotation.y` — building may need 90° / 180° rotation to face right way
- `group.position` offsets — model origin usually isn't at building center

### D. 4D phasing / time slider
Salar's BIM grading work involves Navisworks TimeLiner 4D simulations. He may want a UI slider to scrub through construction phases. This means tagging mesh groups with `userData.phase = 'foundation' | 'structure' | 'envelope' | 'finishes'` and toggling visibility based on slider value.

### E. Multiple floor plans / multiple buildings
Could use multiple image targets in one `.mind` file (`addAnchor(0)`, `addAnchor(1)`, etc.).

### F. Annotations / labels in AR
Floating text labels above rooms (HTML overlays positioned via 3D-to-screen projection, or `TextGeometry`).

### G. Mobile UX polish
Tap-to-place, pinch-to-scale, rotate gestures for cases where the user wants to inspect the model independent of the tracked target.

## Design language (keep consistent)

The landing page uses an **editorial / architectural drawing aesthetic**:
- Background: paper cream `#f4f1ea` with subtle radial gradients
- Ink: near-black `#0a0a0a`
- Accent: terracotta red `#d94f30`
- Headings: Georgia serif with italic accents on key words
- Body & UI: Helvetica Neue, with `letter-spacing: 0.2em` + uppercase for labels (architectural drawing convention)
- Borders and hairlines, not heavy boxes
- Numbered steps in small caps with red numerals

**Don't drift toward generic AI design (purple gradients, Inter font, rounded everything).** This is meant to feel like an architect's deliverable.

## Known limitations / things NOT in scope yet
